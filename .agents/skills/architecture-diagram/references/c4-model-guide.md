# C4 Model Guide for ADT Architecture Diagrams

## C4 Levels

The C4 model defines four levels of abstraction for software architecture:

### Level 1 — System Context

Shows the system under consideration and its relationships with users
and other systems. Use this to communicate with non-technical stakeholders.

**For ADT:** The entire Ansible Dev Tools ecosystem as a single system,
with the Content Author persona, Abbenay (AI daemon/library),
APME (adjacent consumer), and external systems (PyPI, Galaxy, GitHub,
LLM providers).

### Level 2 — Container

Zooms into the system boundary to show the individual deployable units
(applications, services, data stores). Each "container" is separately
deployable.

**For ADT:** Each repo becomes a container. Group them into boundaries:

| Boundary | Contents |
|----------|----------|
| ADT Core Tools | ansible-dev-tools, ansible-lint, ansible-compat, ansible-navigator, ansible-dev-environment, ansible-creator, ansible-sign, molecule |
| ADT Testing & Automation | pytest-ansible, tox-ansible, actions, ansible-content-actions |
| ADT Extensions | vscode-ansible (with language server + MCP sub-components) |
| Community | mkdocs-ansible |
| Downstream Containers | ansible-devtools-container, ansible-devspaces-container |
| Abbenay (AI Daemon) | @abbenay/core, @abbenay/daemon, VS Code ext, Python client |
| APME (Adjacent) | engine, validators, gateway, frontend, Galaxy Proxy |

### Level 3 — Component

Zooms into a single container to show its internal components. Use this
for repos with rich internal architecture.

**Best candidates for L3 in ADT:**

- **vscode-ansible** — language server, MCP server, Python tool
  integrations, extension host, webview panels
- **abbenay** — core library, daemon layer, gRPC server, web dashboard,
  VS Code extension, Python client, proto stubs
- **APME** — engine (parse/annotate/fan-out), validators (native, OPA,
  Ansible, Gitleaks, collection health, dep audit), daemon, remediation
  engine, API gateway, Galaxy Proxy, React frontend

### Level 4 — Code

Class/function level detail. Rarely useful for architecture boards.
Not generated by this skill.

## Relationship Types

| Type | Label Convention | Example |
|------|-----------------|---------|
| Direct dependency | "depends on {package}" | ansible-lint depends on ansible-compat |
| CLI spawning | "spawns {tool}" | vscode-ansible spawns ansible-lint |
| Container packaging | "packages into image" | ADT packaged into devtools-container |
| API consumption | "consumes {API}" | AAP Portal consumes ADT web server |
| Build artifact | "builds" | ansible-dev-tools builds community-ansible-dev-tools-image |
| Test dependency | "tests with" | molecule tests with community.molecule |
| Workflow reuse | "uses reusable workflow" | ansible-lint uses ansible/team-devtools workflows |
| AI escalation | "AI remediation via" | APME escalates to Abbenay |

## External Systems

Shown as `SystemExt` (L1) or `ContainerExt` (L2) — visually distinct:

- **ansible-builder** — not owned by DevTools, but ADT and ade depend on it
- **ansible-runner** — Navigator dependency
- **ansible-core** — underlying engine (ansible-playbook, ansible-galaxy, etc.)
- **community.molecule** — molecule and compat test dependency
- **vscode-yaml** — vscode-ansible extension dependency
- **AAP Portal** — custom Backstage instance for AAP self-service; consumes ADT web server and REST API
- **LLM Provider APIs** — abbenay talks to OpenAI, Anthropic, Gemini, Ollama, etc.
- **PyPI** — where Python packages are published
- **Ansible Galaxy** — where collections are published
- **GitHub** — source code hosting and CI/CD
